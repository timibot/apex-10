# APEX-10: Approved System Rules & Architecture
*(This document is the absolute source of truth for how the inference pipeline, filters, and ticket generator must behave. Do not alter these rules without explicit user approval.)*

## 1. Ticket Generation & Size
* **Dynamic Legs:** The "9 legs" target is strictly **NOT FIXED**. The optimizer must build a ticket of whatever size naturally survives the gates. It should never artificially block or drop an entire ticket simply because it didn't hit a magical 10.0x combined odds threshold.
* **Selection Logic / Tiebreakers:** Sort all qualifying matches by:
  1. **Confidence Level** (5/5 vs 4/5 - descending)
  2. **Model Probability** (descending)
  3. **Highest Odds** (descending)  
  *(Rule: If two games have the same 5/5 confidence and same probability, pick the one with the highest odds).*

## 2. Gate Filters (The Non-Negotiable Rules)
* **Odds Range:** Matches MUST have odds strictly between **1.20 and 1.49**. Any match predicting odds below 1.20 (like 1.13) or above 1.49 must be relentlessly dropped.
* **League Cap (Correlation Gate):** Maximum of **3 matches** per league on any single accumulator ticket. 
* **Confidence Overlay:** We only want mathematically safe matches. The system generally requires **5/5 confidence**, but will also process 4/5 confidence matches if they represent extreme value or belong to a safe pattern. (*Note on League Cap Overrides:* If a league hits its 3-match cap, the system is allowed to swap out a lower-confidence 4/5 match to make room for a golden 5/5 match).

## 3. The Purpose of This Stage (Paper Trading)
* **Current Phase:** The current GitHub Action setup is strictly for **Paper Trading and Data Gathering**. 
* **Objective:** The goal right now is to generate tickets, observe how the AI evaluates the 5 newly integrated leagues (using the 29-feature array), and track the simulated Brier scores over time. We are **not** live trading yet. We are tracking performance, validating that the 1.20–1.49 odds range actually hits, and identifying mathematical weaknesses.

## 4. Odds Architecture & API Fallbacks
* **Odds API Exhaustion:** If the live OddsAPI quota is exhausted ("finished") or returns limit errors, the system MUST immediately alert the user to provide a new API key, rather than failing silently or exclusively relying solely on "fake" (theoretical) odds across the board without warning.
* **Missing Odds Fallback:** For individual games where the Odds API simply does not have the odds available (but the API is otherwise functioning), we use our previously decided method: falling back to **Poisson Distribution Mathematics** to predict the theoretical fair odds.
* **Gate Enforcement:** Even if the odds are generated by Poisson (e.g. 1.13), they are still aggressively subjected to the 1.20–1.49 Gate Filter. Wait for actual bookmaker odds to align.

## 5. Derived Odds (Bypassing Free Tier Limitations)
Some premium markets are not provided directly on the Free Tier of The Odds API. To bypass this, the inference engine **mathematically derives** these premium odds from the standard `H2H` (Home, Draw, Away) odds that we *do* get.
* **Double Chance 1X (Home Win or Draw):**
  * *Calculation:* `Implied Probability = (1 / Home_Odds) + (1 / Draw_Odds)`
  * *Derived Odds:* `1 / Implied Probability`
* **Draw No Bet (DNB Home):**
  * *Calculation:* `DNB Implied Probability = (1 / Home_Odds) / ((1 / Home_Odds) + (1 / Away_Odds))`
  * *Derived Odds:* `1 / DNB Implied Probability`
